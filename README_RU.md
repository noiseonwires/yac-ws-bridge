Экспериментальный вариант туннеля brdige-to-freedom.

По сути дела то же самое, что в бранче main, но вместо своего протокола мульиплексирования и коррекции ошибок использует QUIC завернутый в WS.
Адаптер генерит самоподписанный сертификат при старте, хелпер его не проверяет - не нужно, проверка клиента идет по AUTH_TOKEN.

Конфигурируется точно так же, добавляются параметры

```yaml
quic:
  mtu: 1500         # Размер QUIC пакета (байт). По умолчанию 1200.
  sendWorkers: 2     # Количество параллельных горутин для отправки wsSend.
```
у адаптера и хелпера.

## Оценка пропускной способности

Максимальная пропускная способность ограничена скоростью вызовов wsSend API:

```
throughput = sendWorkers × (1000 / wsSend_latency_ms) × MTU
```

Например, с настройками по умолчанию при латентности wsSend ~30мс:

```
8 × (1000 / 30) × 1200 ≈ 320 КБ/с
```

Чтобы увеличить скорость:
- **Увеличить `mtu`** (например 8192 или 16384) — больше данных в каждом вызове API, не увеличивая нагрузку на API
- **Увеличить `sendWorkers`** (например 16 или 32) — больше параллельных вызовов gRPC, но пропорционально растёт нагрузка на API

Не забудьте передеплоить serverless function перед экспериментами и после.

MAUI-клиента нет, потому что реализация QUIC из .NET, как выяснилось, недоступна на мобильных платформах.
