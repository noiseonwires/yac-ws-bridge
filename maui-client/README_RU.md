# Bridge to Freedom — Android-клиент

GUI-клиент для IP-туннеля Bridge to Freedom. Работает как **helper**: поднимает системный VPN (`VpnService`) и гоняет сырые IP-пакеты через YC API Gateway к адаптеру на другой стороне.

> ⚠️ **Этот клиент лежит в бранче `ip-tun`.** Перед сборкой переключитесь: `git checkout ip-tun`. На других ветках — другие (несовместимые) версии хелпера.

## Для чего это годится (а для чего нет)

Это serverless WebSocket, прикидывающийся L3-туннелем. По ощущениям пропускная способность похожа на старый 2G/EDGE — для обычного веб-сёрфинга, стримов и больших скачиваний **непригодно**.

Зато **прекрасно работает Telegram**: его протокол гоняет мелкие сообщения, легко реконнектится и нечувствителен к задержкам — именно то, что туннель умеет хорошо. Чаты, голосовые сообщения, мелкие картинки/стикеры работают вполне юзабельно.

**Рекомендуемая схема**: включите в приложении **per-app tunneling** и пустите через туннель **только Telegram** (опционально — ещё одно-два нетребовательных к трафику приложения). Остальное оставьте на обычной сети — у вас будет нормальный быстрый интернет для всего обычного и работающий Telegram сверху.

## Поддерживаемые платформы

- **Только Android** (API 26+ / Android 8.0+) — с поддержкой **per-app tunneling**.

IP-туннельная архитектура требует прямого доступа к TUN-устройству, который нельзя добавить чистым managed-MAUI-кодом на других платформах:

- **iOS** требует отдельного `NEPacketTunnelProvider`-таргета в виде системного расширения — за рамками одного MAUI-проекта.
- **Windows** требует загрузки нативной `wintun.dll` — используйте Go-бинарник [`helper`](../adapter-and-helper/cmd/helper).
- **macOS** требует системного расширения с Network Extension entitlement — то же самое, используйте Go-хелпер.

Если нужен десктопный клиент — соберите Go-[`helper`](../adapter-and-helper) и следуйте инструкциям по маршрутизации в корневом [README_RU.md](../README_RU.md#маршрутизация-на-клиенте).

## Требования

- .NET 10 SDK (на момент написания — preview)
- MAUI workload: `dotnet workload install maui`
- Android SDK (ставится вместе с MAUI workload)

## Разрешения

Приложение запрашивает (на установке / в рантайме):

- `BIND_VPN_SERVICE` — нужен для регистрации `BtfVpnService` как системного VPN.
- `FOREGROUND_SERVICE` / `FOREGROUND_SERVICE_DATA_SYNC` — чтобы держать VPN живым в фоне.
- `WAKE_LOCK` — удерживается пока туннель поднят.
- `INTERNET` — для upstream WebSocket.

При первом подключении Android показывает системный диалог «Запрос соединения» («…хочет установить VPN-соединение»). Подтвердите один раз.

## Сборка

```bash
cd maui-client

# Debug
dotnet build -f net10.0-android

# Release APK
dotnet publish -f net10.0-android -c Release
```

APK будет в `bin/Release/net10.0-android/publish/`.

## Использование

1. Введите **Bridge URL** — helper-эндпоинт API Gateway (например `wss://gateway.example.com/_helper`).
2. Введите **Auth Token** — общий секрет, совпадающий с адаптером и Cloud Function.
3. **Tunnel Address** — локальный IP туннеля на устройстве (по умолчанию `10.200.0.2`).
4. **Peer Address** — туннельный IP адаптера (по умолчанию `10.200.0.1`).
5. **MTU** — по умолчанию `1400`. Уменьшите, если сеть рубит крупные пакеты.
6. **Relay mode** — включите, если устройство не может достучаться до gRPC `wsSend` напрямую.
7. **Per-app tunneling** *(настоятельно рекомендуется — см. «Для чего это годится» выше)* — выберите список приложений, которые надо пускать через туннель; всё остальное пойдёт по обычной сети. Для рекомендованной «только Telegram» схемы — выберите здесь только Telegram. Если список пустой, через туннель идут **все** приложения (кроме самого этого приложения — оно исключается автоматически).
8. Нажмите **CONNECT** и подтвердите системный запрос на VPN.

Приложение прописывает default route (`0.0.0.0/0`) через TUN с DNS `1.1.1.1` / `8.8.8.8`. Если настроен per-app tunneling, Android применяет allow-list на уровне ОС — невыбранные приложения вообще не идут в VPN. В любом случае собственный пакет приложения исключается из своего же VPN, чтобы WebSocket не зациклился.

Foreground-service держит постоянное уведомление, пока туннель поднят. **DISCONNECT** останавливает и снимает VPN.

Настройки сохраняются автоматически.

## Избежание петли

Android `VpnService` не позволяет исключать отдельные хосты по маршрутам, поэтому приложение исключает целиком свой пакет. Если вы используете другой VPN/TUN-прокси на этом же устройстве — одновременно активным может быть только один VPN, это ограничение Android.

## Если не работает

- **«VPN permission denied»** — пользователь отказался в системном диалоге. Нажмите CONNECT снова, чтобы повторить запрос.
- **Туннель поднят, но трафика нет** — чаще всего на адаптере не включён `net.ipv4.ip_forward` или нет MASQUERADE-правила. См. корневой [README_RU.md](../README_RU.md#ip-форвардинг-на-сервере-linux). Если используется per-app tunneling — убедитесь, что нужное приложение реально выбрано в списке.
- **Отваливается каждые ~10 минут** — слишком большой `pingIntervalMs`; YC рубит idle WebSocket'ы на 10-й минуте.
- **Telegram работает, а браузер — нет** — так и должно быть. См. «Для чего это годится» в начале.
